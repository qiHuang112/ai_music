# AM-005 搜索歌曲库与歌词稳定性设计

## 目标

1. 搜索页面在搜索页 HTML 返回后立即展示候选状态，不再等待整页逐首校验完成。
2. 完整音频校验保持 fail closed，同时通过最多 3 路有界并发缩短首个可播结果和整页完成时间。
3. 当前搜索的已验证结果组成动态歌曲库，后续分页结果持续追加，并按用户当前播放模式切歌。
4. 搜索歌曲按需解析与边下边播，不因入队而预下载整页。
5. 流媒体歌曲的歌词在播放前绑定，缓存转正后继续执行元数据兜底；详情页不再显示停止按钮。

## 搜索数据流

搜索页请求与候选完整音频校验分为两个阶段：

- 第一阶段只请求并解析搜索页。解析出的候选立即以 `validating` 状态进入控制器和 UI，播放、下载按钮暂时不可用。
- 第二阶段以最多 3 个并发任务执行详情页、播放器 API、HEAD 与 Range 校验。成功候选变为 `ready` 并进入动态歌曲库；失败候选从结果中移除，不能写入正式缓存。
- 第一条成功候选出现后即可播放；页面级加载状态继续到本页所有候选完成。
- 分页沿用滚动到底自动加载。每页成功候选按搜索页原始顺序合并和去重，并追加到当前动态歌曲库。

公共 HTTP 客户端不再对交互搜索 GET 使用固定 `12s` 建连超时和 3 次透明重试。搜索链路使用明确的短预算与最多一次显式瞬时错误重试；单个候选失败不阻塞同页其他候选，也不把整页变成失败。

## 动态搜索歌曲库

控制器维护一个以搜索代次为边界的 `SearchPlaybackSession`：

- 保存查询、已验证候选、当前候选、稳定随机顺序和当前索引。
- 点击任一可播候选时激活当前搜索代次；之后新分页的已验证候选动态追加。
- 顺序模式播放下一项，到末尾停止自动切歌；列表循环在末尾回到第一项；单曲循环重播当前项；随机模式使用稳定随机顺序。
- 页面上一首、下一首、播放结束、通知栏切歌和队列面板点选统一委托给该会话。
- 切到某一候选时才消费已准备的解析结果、创建本地代理并开始边下边播；仅入库或追加分页不会启动音频下载。
- 离开搜索并开始播放收藏、缓存或自建歌单时，清除动态搜索会话，恢复既有真实播放器队列行为。

音频处理器保留当前实际可播放源，同时发布独立的展示队列。搜索会话负责把队列操作转换成候选的懒加载，避免为了让 `just_audio` 看到整页 URI 而提前打开所有代理。

## 歌词与元数据

- 完整音频解析成功后，在等待播放器加载前就写入当前流媒体歌曲的封面和歌词。
- 控制器保留临时流媒体 ID、候选键和解析结果的关联，不再依赖 `cachedTracks` 中是否已有同 ID 的 `currentTrack`。
- 缓存转正后，使用正式 `CachedTrack` 继续执行本地歌词、搜索页歌词和网络兜底 provider；若当前仍是同一播放代次，则原子更新当前歌词与封面。
- 所有异步元数据结果都检查播放代次，上一首晚返回的歌词不能覆盖当前歌曲。
- 搜索页歌词缺失时自动兜底仍可运行；用户手动重试继续支持绕过有效 miss TTL。

## UI 与错误状态

- 校验中的候选显示明确的加载状态且不可点击；校验成功后变为可播放、可下载。
- 当前队列面板展示动态搜索歌曲库和当前项，后续分页追加后数量同步增长。
- 播放详情页和歌词详情页共用的播控行移除停止按钮；停止能力仅保留为内部生命周期和队列清理接口。
- 页面请求失败显示结构化搜索错误；候选级失败静默剔除。若本页无任何成功候选且无下一页，显示无可播放结果。

## 验收与测试

- 正常网络下搜索页解析后立即出现 `validating` 候选，首个成功候选无需等待整页完成。
- 候选校验并发数不超过 3，输出仍按搜索页顺序；失败候选不进入播放库或缓存。
- 交互搜索不会执行 `12s` 建连超时的三次透明重试。
- 播放任一搜索结果后，当前及后续分页成功候选动态追加到队列，四种播放模式均符合定义。
- 队列追加不会预下载未播放歌曲；切歌时才创建对应代理。
- 流媒体歌词在进入详情页时可用，缺失时缓存转正后自动兜底，旧请求不能覆盖新歌。
- 播放详情和歌词详情均无停止按钮。
- Android 小米 10 Pro 验证搜索首屏、分页追加、队列切歌、自动播完、歌词和边下边播 seek；完成后恢复搜狗输入法。

## 非目标

- 不修改搜索源范围或完整音频判定标准。
- 不为搜索结果预下载整页音频。
- 不在本次调整页面整体视觉风格或新增播放模式。
