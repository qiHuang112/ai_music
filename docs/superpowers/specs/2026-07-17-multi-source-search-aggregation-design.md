# AI Music 多源完整音频搜索聚合设计

Created: 2026-07-17
Status: User approved
Workflow: superpowers-v1
Work Type: feature
User Visible: yes

## 背景

当前 `Auto` 搜索虽然已经具备歌曲海、酷我完整音频、BuguYY、FLAC、22a5 等解析器，但分页主链路采用单源短路策略：歌曲海只要返回非空结果，本页就直接结束；只有歌曲海为空或失败时才切换到酷我。因此歌曲海只返回少量候选时，用户只能看到该单一来源的结果。

2026-07-17 低频串行探测显示：

- 酷我搜索和完整 MP3 转换当前可用，适合与歌曲海组成第一阶段聚合源。
- BuguYY 搜索能返回候选，但播放接口当前返回客户端类型限制，不能作为完整可播完成路径。
- 22a5 当前进入人机安全验证页，不能通过绕过验证的方式接入。
- FLAC 源依赖挑战链路且搜索时延较高，暂不进入第一阶段主搜索。

## 用户目标

在线搜索不再被歌曲海单一来源的结果数量限制。用户应尽快看到第一批完整可播歌曲，并在后台校验完成后看到来自其他可靠来源的更多结果；新增结果继续成为当前搜索歌曲库的一部分。

## 方案选择

### 采用：严格校验后的多源动态聚合

歌曲海与酷我完整音频共同参与 `Auto` 搜索。两个来源都必须完成完整音频准入校验，结果才可进入列表和播放队列。

### 不采用：顺序回退

歌曲海非空就停止、为空才搜索酷我，虽然请求量低，但不能解决少结果问题。

### 暂不采用：全部已有解析器同时聚合

BuguYY、FLAC、22a5 当前分别存在客户端限制、挑战时延或人机验证。将它们直接放入主链路会增加搜索耗时和不可播结果，不满足完整音频质量底线。

## 范围

- `Auto` 搜索聚合歌曲海与酷我完整音频。
- 搜索首屏同步聚合设备上已经完整落盘、索引有效且匹配查询的缓存歌曲；本地命中不产生任何网络请求。
- 每个来源独立分页、独立熔断、独立错误状态。
- 严格校验后的动态追加、跨来源去重和搜索播放队列增长。
- Android、小米真机主路径验证，以及公共 Dart 最小匹配测试。

## 2026-07-17 真机反馈修订

- 小米 10 Pro 直连 `www.gequhai.com` 时 TCP 连接超时，而同一请求在走代理的 Mac 上返回 200；验收不得再用 Mac 成功推断手机可用。
- 用户明确否定“每次下拉只增加一首”，要求首屏批量呈现并继续动态追加。因此允许控制器串行预取最多 3 个聚合页，达到 8 首或来源耗尽后立即停止；同一来源只有上一页完成后才能请求下一页。
- 本机完整缓存是搜索歌曲库的正式来源：命中后立即展示、直接本地播放，并与后续在线结果按歌名和歌手去重。
- 用户允许继续评估 BuguYY、FLAC、22a5 等来源，但不得绕过防护、验证码或证书错误，也不得把网盘和试听当成完整音频。每个候选来源只有在真机可达且 HEAD/Range 完整音频校验通过后才能进入聚合列表。
- 额外来源采用低频准入：每来源最多一个搜索任务在途，403、429、防护页或安全验证立即熔断 2 分钟，不做密集重试。

## 非目标

- 不接入 PREVIEW、网盘、HTML 页面或浏览器专用播放地址。
- 不绕过验证码、防护页或第三方安全验证。
- 不把 BuguYY、FLAC、22a5 加入第一阶段 `Auto` 主搜索。
- 不新增来源选择 UI；设置页继续只暴露 `Auto`。
- 不改变现有播放模式、缓存转正、歌词恢复和下载管理规则。

## 架构

### 多源搜索协调器

在数据层增加独立的多源分页协调器，由 `RemoteMusicResolver` 的 `Auto` 路径调用。协调器只负责：

- 启动当前页允许参与的来源流；
- 合并各来源已校验候选；
- 维护来源完成、失败、熔断和分页耗尽状态；
- 输出稳定、去重后的 `MusicSearchProgress` 快照。

歌曲海和酷我解析器继续负责各自协议、匹配门槛、HEAD/Range 校验和 prepared resolution。协调器不复制来源协议，也不放宽来源内部信任门槛。

### 并发与请求压力

- 歌曲海和酷我在同一页开始时并行启动，任一来源最先产出已校验结果即可先显示；酷我不得等待歌曲海超时后才开始。
- 歌曲海自身请求量和验证并发不高于当前实现；新增酷我请求不增加歌曲海域名的请求频率。
- 每个来源最多保留现有 2 个媒体校验 worker，总跨来源媒体校验并发上限为 4。
- 同一查询同一来源的下一页只能在上一页完成后启动。
- 不无限连续预取。首屏最多串行预取 3 个聚合页并在达到 8 首后停止；之后只有用户接近列表底部触发一次 `loadMore`，才请求各未耗尽来源的下一页。
- 新查询通过 request id 使旧查询结果失效；旧流晚返回时不得污染新列表。

## 数据流

1. 用户提交查询，控制器清空上一查询页状态并创建新的搜索会话。
2. `Auto` 协调器并行启动歌曲海第 1 页和酷我第 1 页。
3. 来源内部先输出 validating 状态，但 UI 只把最终 `clientReady` 的完整音频候选视为可见结果。
4. 任一来源产出已校验候选后，协调器与当前快照去重并立即输出。
5. 后续来源完成校验时，只在列表尾部追加新的唯一候选；已显示行不重新排序，避免滚动位置和点击目标跳动。
6. `MusicController` 将新候选追加到当前 `SearchPlaybackSession`，并更新 media session 队列。
7. 未播放候选只做轻量媒体准入校验，不创建 progressive session、不下载音频、不写正式缓存。
8. 用户滚动到底部时，协调器只请求仍声明 `hasNextPage` 的来源。

## 准入规则

候选必须同时满足以下条件才能显示：

- 来源匹配门槛通过，不是低置信歌名或歌手误匹配；
- 媒体类型为 `directAudio`；
- `canCacheAudio == true`；
- `clientReady == true`；
- HEAD 返回音频类型和可接受的长度；
- Range 返回可解析且总长一致的 206；
- 不是 PREVIEW、HTML、防护页、外部网盘或挑战页。

任一条件失败都只记录结构化 `SourceAttempt`，候选不得进入可见列表、播放队列或正式缓存。

## 去重与顺序

- 去重键使用现有标准化能力生成的“歌名 + 歌手”。统一大小写、空白、常见分隔符和已支持的转义文本。
- 不删除版本信息。原版、现场版、DJ 版、伴奏等标题仍不同的内容保留为独立歌曲。
- 同一去重键第一个通过严格校验的候选保留；后到重复项不替换已显示行，也不导致列表重排。
- 去重集合覆盖当前查询的所有已加载页，防止跨页重复。
- 每个来源内部保留原有置信度顺序；跨来源结果按“首次校验完成后追加”排序。

## 分页

- 页码按来源独立维护，不假设歌曲海与酷我的同一页包含等量结果。
- 某来源返回 `hasNextPage=false` 后，后续 `loadMore` 不再请求该来源。
- 聚合页的 `hasNextPage` 只要任一来源仍有下一页即为 true。
- 某来源当前页失败时不擅自跳到下一页；用户后续重试只重试失败来源的当前页。
- 动态追加的歌曲同步进入当前搜索歌曲库，但不改变用户当前正在播放的歌曲和播放位置。

## 错误与熔断

- 每个来源独立记录成功、空结果、超时、403、429、防护页和协议错误。
- 单来源失败且另一来源有有效结果时，继续展示成功结果，不显示“没有在线结果”。
- 403、429 或 defender/challenge 命中后，对对应来源开启 2 分钟进程内熔断；熔断期间不再请求该来源，但其他来源正常工作。
- 两个来源都完成且没有有效候选时，才显示空结果。
- 两个来源都失败时，显示聚合后的用户友好错误，并保留结构化来源失败链用于日志和导出诊断。
- 失败、取消和超时不得创建正式缓存记录。

## UI 行为

- 不新增页面、筛选器或来源设置。
- 首批结果出现后立即结束整页阻塞式 loading，后续来源校验使用列表底部的追加加载状态。
- 每条结果保留已有来源/质量标识。
- 单来源失败但仍有结果时，不使用全页错误遮挡列表；失败详情只进入可重试的页尾状态和诊断日志。
- 新增候选进入搜索歌曲库后，上一首、下一首、随机、列表循环和单曲循环继续由当前播放模式控制。

## 测试设计

### 数据层

- 歌曲海先完成时先输出歌曲海，酷我完成后追加唯一结果。
- 酷我先完成时不等待歌曲海超时，先输出酷我有效结果。
- 跨来源和跨页同曲只保留第一个严格校验通过的候选。
- 一个来源 403/429/defender 后熔断，第二次同进程搜索不再请求该来源，另一来源仍成功。
- 一个来源空结果或失败不覆盖另一个来源的成功结果。
- 两个来源都失败时输出完整结构化失败链。
- 已耗尽来源不会被下一页再次请求。

### 控制器与播放队列

- 动态候选追加后，搜索歌曲库和 media session 队列同步增长。
- 当前歌曲、播放位置和播放模式不因追加结果改变。
- 未播放候选不启动 progressive cache、不写 part 文件、不写正式缓存。
- 新查询开始后，旧查询晚返回结果被丢弃。

### UI 与真机

- 首批结果可交互时不再显示整页等待。
- 列表尾部追加不重排已有行、不改变滚动位置。
- 小米 10 Pro 搜索 `周杰伦`、`Angel` 和一个歌曲名样例，证明列表含两个来源的严格可播结果。
- 从两个不同来源各播放一首，覆盖完整播放、动态队列、seek、缓存转正和失败不污染缓存。

## 验收标准

1. `Auto` 搜索同一查询能够动态展示歌曲海和酷我两个来源的严格可播结果，且已有行不跳动。
2. 任何可见结果都通过完整音频 HEAD/Range 校验，PREVIEW、HTML、网盘、防护页和低置信候选不可见。
3. 单个来源失败或熔断时，另一个来源仍能返回、分页和播放，不错误显示全局无结果。
4. 动态追加结果进入当前搜索歌曲库并遵循现有播放模式，未播放歌曲不预下载。
5. 定向测试、全量测试、analyze、diff-check、APK/设备包 SHA 和小米 10 Pro 双来源真机证据全部通过后才可提交负责人 review。

## 风险与控制

- 多源并行会增加总请求量，但不增加歌曲海单域并发；通过每源 worker 上限、按需翻页和来源熔断控制压力。
- 严格校验会使后续结果逐步出现，但不会用不可播候选换取虚假数量。
- 酷我或歌曲海协议随时可能变化；失败必须局部化并保留结构化证据，不能把空结果伪装成成功。
- 跨来源去重可能遇到艺人别名或版本标题差异；第一阶段保持保守去重，宁可保留疑似不同版本，不误删真实歌曲。

## 后续来源准入

BuguYY、FLAC、22a5 只有同时满足以下条件后，才能通过独立任务加入聚合器：

- 低频脚本和客户端都能稳定复现搜索与完整音频解析；
- 不依赖绕过验证码、防护或浏览器人工状态；
- 与歌曲海、酷我使用相同的匹配、HEAD、Range、缓存和失败门槛；
- 有单来源熔断、失败不污染缓存和真实设备证据。
